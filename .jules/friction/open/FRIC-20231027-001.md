## Problem
The documentation (`ROADMAP.md`, `docs/PRODUCT.md`, `docs/implementation-plan.md`) incorrectly referenced a shipped `tokmd serve` command for MCP Server functionality. While the PR attempt to correct this by recasting it to the shipped `tokmd tools` schema generation was rejected (in favor of a different PR #1588), there is still unclarity/drift between the current shipped state (`tokmd tools` for schemas) and the future planned state (`tokmd serve` for MCP resources).

## Evidence
- `tokmd serve` is not a shipped command.
- `tokmd tools` exists and generates schema definitions.
- The PR recasting MCP Server to `tokmd tools` was closed in favor of #1588, stating: "`tokmd serve` / MCP server resources remain future work, so this PR's MCP-server recast should not land as written".
- See `ROADMAP.md` and `docs/implementation-plan.md` which still contain references to Phase 6 and `tokmd serve`.

## Why it matters
Documentation drift regarding future vs shipped capabilities (especially around a highly visible feature like MCP integration) creates confusion for users and contributors. The correct boundaries and current state need to be accurately reflected in the truth sources.

## Done when
- [ ] Documentation accurately reflects that `tokmd tools` is shipped for schema generation.
- [ ] Documentation accurately reflects that `tokmd serve` (the MCP Server) is explicitly planned future work and not currently available.
