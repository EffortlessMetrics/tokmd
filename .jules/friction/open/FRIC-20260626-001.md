# Friction Item

id: FRIC-20260626-001
persona: fuzzer
style: Prover
shard: interfaces
status: open

## Problem
In the FFI `run_json` boundary, which serves Python and Node wrappers, the configuration payload logic extracts nested setting objects using `.unwrap_or(args)`. If a user incorrectly provides a non-object (e.g. `{"lang": "bogus"}`), the code fails to parse the inner block and silently falls back to using `args` as the context object. This misses explicit validation on structure boundaries and masks invalid types from users.

## Evidence
- `crates/tokmd-core/src/ffi/settings_parse.rs`
- Running a test payload like `{"lang": "string value"}` against the FFI entrypoint does not trigger an error on the `lang` field's type.

## Why it matters
Input hardening in FFI boundaries is critical to prevent silent logic errors or unexpected behavior across language bindings. Failing to validate nested configuration shapes forces users to debug opaque failures.

## Done when
- [ ] Replace `.unwrap_or(args)` with a strict object-checking helper (`nested_arg_object`) across mode settings parsers.
- [ ] Add an integration test that asserts passing an invalid scalar/string to a nested block configuration returns an explicit JSON error payload rather than succeeding or defaulting silently.
