## Target
- Hot loop performance issue in `crates/tokmd-model/src/lib.rs`, `crates/tokmd-analysis-content/src/content.rs`, and `crates/tokmd-analysis-derived/src/lib.rs` where `.entry(row.lang.clone())` forces an allocation of a string just for a map lookup and insertion when iterating large number of files.

## Proof Method
- Structural proof: String allocation eliminated from hot loops iterating over rows using borrowed string `&str` for the BTreeMap keys.

## Options A/B
### Option A (recommended)
- Use `&str` instead of `String` for `BTreeMap` keys in `by_lang`, `by_module`, `by_size` across several crates.
- Pros: Eliminates heavy string allocations inside large iterators during analysis generation.
- Cons: Needs mapping back to `String` at the collection stage, which is done only once per distinct string.

### Option B
- Keep `String` and just accept the performance hit.
- Pros: Easier to implement.
- Cons: Sub-optimal performance on big repositories.

## Decision
- Option A. It avoids `O(N)` cloning across thousands of rows and fits well within Rust's zero-cost borrowing model where the rows live longer than the scope of the maps doing the reduction.
