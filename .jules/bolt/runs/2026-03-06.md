## Lane
- Scout discovery: Performance bottleneck in `tokmd-analysis-format` and `tokmd/src/commands/context.rs` where intermediate string allocations are used extensively via `.push_str(&format!(...))` instead of writing to a pre-allocated string using `std::fmt::Write`.

## Target
- Target: Replace `.push_str(&format!(...))` patterns with `writeln!` and `write!` macros leveraging `std::fmt::Write` in `tokmd-analysis-format` (`render_md` and `render_xml`) and `tokmd` (`commands/context.rs` `format_list_output`). Add `with_capacity` where possible.

## Options
### Option A (recommended)
- Replace `.push_str(&format!(...))` with `std::fmt::Write` trait's `write!` and `writeln!` macros on `String` buffers.
- Pre-allocate the string using `String::with_capacity`.
- Trade-offs: Structure: Requires mutability and `Write` trait in scope, but improves performance by avoiding intermediate allocations. Velocity: Quick to implement, easily scalable to other crates.

### Option B
- Continue using `.push_str` but manually format the strings or use `itoa` or similar crates for numbers.
- Trade-offs: Can be more verbose and harder to maintain, less idiomatic than `std::fmt::Write`.

## Decision
Option A. It's the standard, idiomatic Rust way to build strings without unnecessary heap allocations for intermediate string representations.

## Proof Method
- Structural proof: Replacing `.push_str(&format!(...))` eliminates intermediate O(N) heap allocations for every formatted line.
- Benchmarking: We will use cargo test timing or a simple measurement if available. (We'll focus on structural work reduction as proof).
