# Bolt Run Log
Read: CI gates (cargo build, test, fmt, clippy) and docs.
Lane: Scout discovery
Target: `crates/tokmd-analysis-effort/src/delta.rs: build_delta` uses multiple `BTreeSet<String>` and `BTreeMap<String, (String, String)>` leading to many unnecessary `.clone()` calls during `export.rows` iteration and git delta path processing.

## Options Considered

### Option A (recommended)
- What: Refactor `build_delta` to use `BTreeSet<&str>` and `BTreeMap<&str, (&str, &str)>` to eliminate allocation inside the `for row in &export.rows` loop and `for path in changed.keys()` loop. The inputs `&export.rows` already own the strings so we can just reference them.
- Fit: High structural gain, no dependencies added, purely mechanical borrowing instead of cloning.
- Trade-offs: Requires a bit more explicit lifetimes or `as_str()` but safe since `export` and `changed` outlive the local sets used purely to compute counts.

### Option B
- What: Use `FxHashSet` and `FxHashMap` instead of BTree structures.
- Fit: Faster lookups, but does not eliminate string cloning, just improves insertion speed. Breaks determinism guarantees from BTree structures if order matters (it doesn't seem to for just calling `.len()`, but BTree is safe default).

## Decision
Option A. Eliminates string allocations inside the loops entirely.

## Proof
Structural. We are replacing multiple `.clone()` calls per row/changed-path with zero-allocation `&str` references. For a repo with 10k files, this eliminates 30k+ string allocations on the delta path.
