# Decision

## Option A (recommended)
Add practical `after_help` examples to the CLI command parsers for `diff`, `gate`, `run`, `export`, and `tools` (aligning with `analyze`, `context`, and others which already have them). Then regenerate `docs/reference-cli.md` with `cargo xtask docs --update`.

- **Why it fits**: It directly fulfills the ROADMAP.md goal "CLI help examples - Add practical examples to command help for analyze, diff, context, gate, cockpit, handoff, run, and export." It also ensures the CLI reference is consistent.
- **Trade-offs**: Structural. Adds boilerplate to parser files but significantly improves the developer experience.

## Option B
Do not modify the CLI parsers, but only append examples to `docs/reference-cli.md` manually.

- **When to choose**: If the help text is meant to be hand-maintained separately from the code.
- **Trade-offs**: Bad governance. The CLI docs are generated by `cargo xtask docs`, so manual edits will be overwritten, and users running `tokmd diff --help` would not see the examples.

## Decision
Option A. I will add the `#[command(after_help = "...")]` attributes to the CLI arg structs and update the docs via `cargo xtask docs --update`.
