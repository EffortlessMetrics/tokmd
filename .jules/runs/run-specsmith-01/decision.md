# Decision

## Option A (recommended)
Write a new integration test explicitly validating that when an invalid/unrecognized subcommand is passed, `tokmd` correctly exits with a non-zero status code and prints a helpful message to stderr (not stdout). This locks in a critical CLI interface edge case regression. We will add a new test file or append to an existing BDD file in `crates/tokmd/tests/` to strengthen edge case coverage without touching implementation logic.

## Option B
Find an undocumented edge case in configuration schema merging and tighten its parsing.

## Chosen Option
Option A fits the Specsmith persona perfectly, addressing "missing BDD/integration coverage for an important path" and "edge-case regression not locked in by tests". The current CLI error path tests (`cli_e2e_w58.rs`, `error_handling_w70.rs`, etc.) all verify that `stderr` is not empty when an unknown subcommand is given, but they don't explicitly verify the exact error message text or that it prints a help suggestion matching the `clap` output for unknown subcommands vs missing paths. Actually, `clap` naturally handles unknown subcommands with an error, but looking at my test of `tokmd frobnicate`, it actually prints:
```
Error: Path not found: frobnicate

Hints:
- Verify the input path exists and is readable.
- Use an absolute path to avoid working-directory confusion.
- If this was meant to be a subcommand, it is not recognized. Use `tokmd --help`.
```
It is currently being treated as a **path argument to the default `lang` subcommand** rather than a clap unrecognized subcommand. The error message is currently generated by `tokmd`'s path validation logic, rather than clap's subcommand parsing. This is because `paths` is an optional positional argument on `CliLangArgs`, and the default subcommand is `Lang`.

I will write a specific integration test (or patch an existing one) to verify that this *exact* error hint behavior is preserved and locked in, since it represents a custom, carefully designed error message for the ambiguous "is it a path or a typoed subcommand" edge case. I'll create a dedicated learning PR packet focusing on this CLI error path behavior.
