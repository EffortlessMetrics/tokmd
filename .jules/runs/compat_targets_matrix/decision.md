## 🧭 Options considered

### Option A (recommended)
- Replace platform-dependent `String.prototype.localeCompare` in `crates/tokmd-format/src/analysis/templates/report.html` with explicit strict Unicode code unit comparison (`a < b ? -1 : a > b ? 1 : 0`).
- **Why it fits this repo and shard**: The shard emphasizes target-specific behavior, and the memory indicates determinism drift from JS's `localeCompare` should be replaced with strict unicode sorting to match Rust. The analysis HTML format generated by `tokmd-format` implements string sorting via JS and is thus susceptible to platform drift across different browsers or locales.
- **Trade-offs**:
  - **Structure**: Increases determinism and guarantees behavior aligns with Rust's lexicographical order and BTreeMap behavior.
  - **Velocity**: Minimal code change, high impact on preventing bug reports regarding indeterministic reports across browsers.
  - **Governance**: Complies strictly with anti-drift rule.

### Option B
- Modify the Rust core to supply pre-sorted HTML rows to the frontend so JS sorting relies only on row indices, or bring in an external sorting dependency to standardise locale sorting.
- **When to choose it instead**: If strict Unicode code unit sorting was inappropriate for string tables.
- **Trade-offs**: Heavily alters structure, adds complexity or bloat without a compelling reason, and pushes logic across language boundaries in ways that might unnecessarily complicate templates.

## ✅ Decision
Option A was chosen. It directly fixes the determinism drift in `report.html` caused by `localeCompare()` while keeping the change highly focused and adhering to memory constraints about strict Unicode code-unit comparisons matching Rust natively.
