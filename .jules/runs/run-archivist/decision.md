# Decision

## Context
As the Archivist, my goal is to improve Jules itself by consolidating run packets and summarizing them into generated indexes/rollups.
I noticed that `.jules/bin/build_index.py` only aggregates active runs from `.jules/runs/`. However, historical run packets exist in version control under `.jules/docs/` (e.g., `ledger.json` and run markdown files). Memory states that these historical runs must be preserved as primary truth and tooling like `build_index.py` should be updated to aggregate them natively instead.

## Options Considered

### Option A (recommended)
Update `.jules/bin/build_index.py` to also read `.jules/docs/ledger.json` and append these historical runs to the `RUNS_ROLLUP.md` index. We can infer `gates_run` from `receipts` and potentially `shard` from `lane`, while setting status to `success` (as these are completed historical runs).
- **Why it fits:** It directly satisfies the prompt's instruction to summarize per-run packets into generated indexes/rollups and the memory constraint to aggregate historical runs natively.
- **Trade-offs:**
  - Structure: Improves completeness of the index.
  - Velocity: Low implementation effort, high reward.
  - Governance: Aligns with keeping historical truth untouched while improving observability.

### Option B
Migrate the historical runs from `.jules/docs/` into `.jules/runs/` format.
- **Why it fits:** It standardizes the format of all runs.
- **When to choose it:** If we want to strictly enforce one packet format globally.
- **Trade-offs:** Violates the memory constraint which explicitly forbids deleting or rewriting historical run packets already tracked in version control, and requires more effort.

## Decision
**Option A**. It safely aggregates historical runs without violating the constraint against rewriting history, improving the completeness of the index generated by `build_index.py`.
