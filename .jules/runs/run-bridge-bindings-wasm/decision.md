# Decision

## Inspected
- `crates/tokmd-core/src/ffi/` (JSON FFI entrypoint)
- `crates/tokmd-wasm/src/lib.rs` (WASM bindings)
- `web/runner/messages.js` (Browser runner message validation)
- `web/runner/runtime.js` (Browser runner runtime)
- `web/runner/messages.test.mjs` (Browser runner validation tests)

## Findings
The browser runner implements custom pre-validation of run message arguments in `web/runner/messages.js` via functions like `isInMemoryInput`, `resolveRunInputs`, and `isRunArgsForMode`. These duplicate logic from `crates/tokmd-core/src/ffi/inputs.rs` and `crates/tokmd-core/src/ffi/settings_parse.rs`.

If the JavaScript `isRunMessage` fails, `handleRunnerMessage` immediately returns an `invalid_message` error rather than delegating to the Rust core (`tokmd-wasm`) which has high-context, standardized `invalid_settings` errors with specific messages explaining exactly *why* a setting or input is malformed (e.g., "a non-empty relative file path", "cannot be combined with in-memory inputs").

This is a direct violation of the memory note:
> In the `tokmd` project, JS/UI bindings (e.g., `web/runner`) should delegate settings and argument validation to the Rust core (`tokmd-wasm`) to preserve high-context `invalid_settings` errors, rather than pre-validating and returning generic errors.

## Options Considered

### Option A: Remove redundant pre-validation in web/runner and delegate to WASM (Recommended)
- **What it is:** Simplify `isRunMessage` and related validation in `web/runner/messages.js` to only check the outer protocol envelope (`type`, `requestId`, `mode`), and blindly pass the `args` payload to `tokmd-wasm`.
- **Why it fits:** It eliminates drift between JS and Rust validation rules and ensures users get high-quality `invalid_settings` errors generated by the core Rust codebase instead of the generic `invalid_message` error.
- **Trade-offs:**
    - Structure: Much better. Single source of truth for validation.
    - Velocity: Faster future updates, no need to sync validation logic.
    - Governance: N/A.

### Option B: Synchronize JS validation to match Rust exactly
- **What it is:** Update `messages.js` to emit `invalid_settings` errors and perfectly mirror all checks in `inputs.rs` (paths, lengths, drive letters, traversals).
- **When to choose it:** If we cannot trust the runner layer at all and want to fail fast before WASM invocation.
- **Trade-offs:** Extremely brittle. Will inevitably drift again. Maintenance burden is doubled.

## Decision
Option A. I will remove the duplicate argument validation from `web/runner/messages.js` and allow the Rust core to handle it, preserving the high-quality error messages.
