## 💡 Summary
Included completed historical friction items in the generated `FRICTION_ROLLUP.md`.

## 🎯 Why
The friction rollup index generated by `cargo xtask jules-index` previously only read from `.jules/friction/open/`. Memory guidelines instruct that the generator should parse both `.jules/friction/open/` and `.jules/friction/done/`. This change repairs the index generator so historical context is not lost from the summary table when friction items are completed.

## 🔎 Evidence
- File path: `xtask/src/tasks/jules_index.rs`
- Observed behavior: `write_or_check_friction_rollup` only queries `.jules/friction/open/` and misses items in `.jules/friction/done/`.
- Receipt: `cat .jules/index/generated/FRICTION_ROLLUP.md` shows the table now includes completed items.

## 🧭 Options considered
### Option A (recommended)
- Modify `xtask/src/tasks/jules_index.rs` to collect from both the `open` and `done` directories.
- Fits the `workspace-wide` shard and Archivist persona by consolidating learnings and run scaffolding into shared indexes.
- Trade-offs: Structure (makes the output table comprehensive), Velocity (simple local change in the xtask), Governance (restores accurate visibility into closed friction items for repo maintainers).

### Option B
- Write a secondary manual index.
- When to choose: If xtask tools were unable to build dynamic output.
- Trade-offs: Poor velocity and governance due to lack of automation.

## ✅ Decision
Option A. It's the most effective way to address the target and correctly aligns with the indexing system instructions.
