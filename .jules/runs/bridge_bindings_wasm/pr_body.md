## 💡 Summary
This PR fixes cross-interface drift between the Rust core FFI and the browser runner. Specifically, the browser runner's JS validation now supports receiving in-memory inputs nested under the `scan` object (`args.scan.inputs`), aligning it with the Rust core FFI behavior.

## 🎯 Why
The Rust core FFI (`crates/tokmd-core/src/ffi.rs`) supports providing in-memory inputs either at the root level (`args.inputs`) or nested within a scan object (`args.scan.inputs`), explicitly rejecting inputs if provided in both locations simultaneously. However, the browser runner's message validation (`web/runner/messages.js`) and worker stub implementation only recognized the root-level `args.inputs`. This caused a drift where valid payloads generated by external clients or matching core behavior would be rejected by the browser runner.

## 🔎 Evidence
- `crates/tokmd-core/src/ffi.rs`: The `parse_in_memory_inputs` function checks both `args.get("inputs")` and `scan.get("inputs")` and returns an error if both are present.
- `web/runner/messages.js`: The `isRunArgsForMode` function was checking only `args.inputs` and rejecting the payload if it wasn't present, even if `args.scan.inputs` was.

## 🧭 Options considered
### Option A (recommended)
- **What it is:** Update `web/runner/messages.js` to look for inputs at both `args.inputs` and `args.scan.inputs`, enforcing mutual exclusivity. Update the worker stub implementation to support both locations.
- **Why it fits this repo and shard:** This matches the behavior defined in the Rust core FFI without modifying the source of truth, thus keeping the browser runner in sync with core logic. It fits perfectly within the `bindings-targets` shard.
- **Trade-offs:**
  - Structure: Aligns the message schema natively.
  - Velocity: Low complexity, purely isolated to the JS wrapper.
  - Governance: Non-breaking change that broadens payload support safely.

### Option B
- **What it is:** Modify the core FFI to enforce only one valid input location (e.g., removing support for `scan.inputs`).
- **When to choose it instead:** If there was a consensus that multiple input locations create unwanted API surface complexity.
- **Trade-offs:** This would be a breaking change to the core FFI which might affect other bindings that already rely on the `scan.inputs` structure.

## ✅ Decision
Option A was chosen. It bridges the gap between the JS wrapper and the Rust core FFI by securely adopting the multiple input location feature already present in the core.

## 🧱 Changes made (SRP)
- `web/runner/messages.js`: Modified `isRunArgsForMode` to extract inputs from either `args.inputs` or `args.scan.inputs`, ensuring both aren't passed simultaneously. Added `scan` to the allowed payload keys.
- `web/runner/messages.test.mjs`: Added tests to ensure nested `scan.inputs` payloads correctly satisfy `isRunMessage`.
- `web/runner/worker.js`: Updated the mapping logic within the worker stub to fallback to `args.scan.inputs` if `args.inputs` isn't provided.
- `web/runner/worker.test.mjs`: Corresponding updates to tests to match the changes made to `web/runner/worker.js`.
