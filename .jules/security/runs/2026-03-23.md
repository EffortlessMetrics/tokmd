# Sentinel Run: 2026-03-23

**Run ID**: 20260323T121016Z
**Lane**: scout discovery
**Target**: Burn down `.unwrap()` in proptest collections (`first()`, `last()`)

## Findings
Multiple `proptest!` blocks use `.first().unwrap()` and `.last().unwrap()` on collections where the size is constrained to be non-empty (e.g. `prop::collection::vec(..., 1..50)`). This contradicts our "Full quality stance: treat unwrap/expect/panic as candidates everywhere" and violates the specific memory instruction:
> When removing .unwrap() from first() or last() calls on Rust collections in proptest! blocks where the collection size is strictly constrained to be non-empty (e.g., vec(..., 1..100)), prefer using .copied().unwrap_or(0) or .unwrap_or(&0) rather than implicit bounds checking (e.g., [0]) or verbose match blocks.

## Options
### Option A (recommended)
- **What it is**: Refactor `unwrap()` calls on `first()` and `last()` in constrained `proptest!` arrays to use `.copied().unwrap_or(0)` or `.unwrap_or(&0)` (for primitives) or fallback values for structs.
- **Why it fits this repo**: Specifically addresses the quality standard memory rule regarding panic burn-downs in tests while adhering to the tight-scoped lane constraints.
- **Trade-offs**: Structure (improved robustness) / Velocity (minor fix time) / Governance (aligns with strict repo policy).

### Option B
- **What it is**: Skip the test-level panic burn down and only focus on production code panics.
- **When to choose it instead**: If the codebase policy explicitly allowed `unwrap()` in tests.
- **Trade-offs**: We would violate the "Full quality stance" which explicitly mentions "(including tests)".

## Decision
Choosing Option A to eliminate test fragility and align with the exact memory guidelines provided for `tokmd`.
