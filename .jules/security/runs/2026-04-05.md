## Findings
The `tokmd-ffi-envelope` crate acts as a boundary parser for JSON data coming from `tokmd_core::ffi::run_json`. In its unit tests and doc examples, it uses `.unwrap()` heavily.
Since this library is meant to be resilient, and as part of the overall roadmap to achieve a panic-free foundation (burning down `.unwrap()`), replacing generic `.unwrap()` with specific context via `.expect()` strengthens the code against hard-to-debug crashes and provides explicit context on what invariant is assumed to hold. This is a solid DX and security/reliability improvement in the test/example suite of a boundary/FFI parsing crate.

## Options Considered

### Option A (Recommended)
Replace all instances of `.unwrap()` in `crates/tokmd-ffi-envelope/src/lib.rs` (in doc tests and unit tests) with context-specific `.expect("...")` messages.
- **Why it fits:** Aligns with the Sentinel directive to burn down `unwrap` globally, even in tests, and ensures boundary code panics provide actionable context.
- **Trade-offs:** Minimal structural change, high reliability signal. Only modifies tests/examples so low risk.

### Option B
Change test signatures to return `anyhow::Result<()>` and use `?` instead of `.unwrap()`.
- **Why it fits:** The idiomatic Rust way to handle errors in tests without panicking.
- **Trade-offs:** Can complicate macro-based tests or doc examples which are often best kept simple to show typical usage. `.expect` is often preferred for doc examples to keep them concise while documenting the expected invariant.

## Decision
**Option A.** Replacing with `.expect()` provides clear invariants for doc tests and unit tests while strictly following the explicit instructions in `ROADMAP.md` and memory constraints.
