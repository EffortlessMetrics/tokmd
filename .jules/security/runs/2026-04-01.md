## Options Considered

### Option A: Refactor `cli_tools_e2e.rs` to use `anyhow::Result<()>` and `?`
- What it is: Change test function signatures to return `anyhow::Result<()>` and replace `.unwrap()` and `.expect()` calls with the `?` operator. For custom assertions that need `.expect()`, provide more highly descriptive messages or use `anyhow::Context`.
- Why it fits this repo: Aligns with the full quality stance to treat `unwrap/expect/panic` as candidates everywhere, including tests.
- Trade-offs: Requires a minor refactor of test functions, but significantly improves error handling and readability.

### Option B: Replace `.unwrap()` with highly descriptive `.expect()`
- What it is: Keep test functions returning `()`, but replace every `.unwrap()` with a very specific `.expect("why this is safe")`.
- When to choose it instead: If refactoring to `Result` is too noisy or breaks macro-based test runners.
- Trade-offs: Still uses panicking methods, which the policy aims to burn down.

## Decision
**Option A**. Refactoring to return `anyhow::Result<()>` and using `?` is the most robust way to burn down panics in tests, adhering to the "full quality stance" mandate.

## Implementation Plan
1. Edit `crates/tokmd/tests/cli_tools_e2e.rs` to change `fn <test_name>()` to `fn <test_name>() -> anyhow::Result<()>`.
2. Replace `.unwrap()` with `?` or `context("...")?` where appropriate.
3. Replace generic `.expect("...")` with `context("...")?` where appropriate.
4. Add `Ok(())` at the end of each test.
