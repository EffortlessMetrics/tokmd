# Sentinel Run 2025-03-12

Lane: Scout
Target: tokmd-core::ffi::tests

## Findings
I discovered multiple instances of `.unwrap()` and `.expect()` calls within the `tokmd-core::ffi::tests` module. Since the repository mandates a full quality stance treating panics as candidates for burn-down everywhere, including tests, I will refactor these tests to use the `?` operator and return `Result<(), Box<dyn std::error::Error>>`.

## Options considered

### Option A (recommended)
- Refactor test signatures to return `Result<(), Box<dyn std::error::Error>>`
- Replace `.expect("json parse")`, `.expect("message string")`, etc. with the `?` operator.
- Replace `.as_str().expect(...)` with `.as_str().ok_or("err")?`.
- This adheres strictly to the quality goal of eliminating panics even in test suites.

### Option B
- Continue allowing panics in tests, limiting refactors strictly to production code.
- This fails to meet the repo's specific requirement for a full quality stance and burndown of `unwrap`/`expect` in tests.

## Decision
I will implement **Option A** to eliminate panicking methods in `tokmd-core::ffi::tests`, returning standard library `Result`s and utilizing `?`.
