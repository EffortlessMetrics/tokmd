## Findings
Multiple instances of `unwrap()` and `unwrap_err()` are used throughout the tests in `crates/tokmd-ffi-envelope/tests/` (specifically `schema_compliance_w53.rs` and `deep_w69.rs`). According to the full quality stance rule ("treat unwrap/expect/panic as candidates everywhere (including tests). Burn-down is the goal."), we should replace these panics with proper `Result` propagation (`?` operator).

## Options Considered

### Option A (recommended)
- **What it is:** Refactor test signatures from `fn test_name() { ... }` to `fn test_name() -> Result<(), Box<dyn std::error::Error>> { ... }` and replace `.unwrap()`/`.unwrap_err()` with `?` and explicit asserts on errors.
- **Why it fits this repo:** Moves test failure handling from ungraceful thread panics to structured `Result`-based error propagation, aligning with Rust best practices and the Sentinel full quality stance.
- **Trade-offs:** Slightly longer test signatures. More robust failures.

### Option B
- **What it is:** Change `unwrap()` calls to `expect("clear error message")`.
- **When to choose it instead:** When the type does not easily convert to a standard `Error`, or changing the test signature is problematic.
- **Trade-offs:** Better error messages than unwrap, but still panics.

## Decision
**Option A**. Replacing `unwrap` with `?` is the preferred way to burn down panics.
